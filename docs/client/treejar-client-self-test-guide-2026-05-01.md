# TreeJar AI Sales Assistant

## Client Self-Test Pack

Дата: 1 мая 2026  
Формат: controlled demo для заказчика и менеджеров TreeJar  
Среда: production admin group в Telegram и защищённый `/dashboard/`

---

## Что Вы Проверяете

TreeJar Assistant уже не просто “чат-бот”. Это операционный sales assistant, который:

- понимает запрос клиента в WhatsApp;
- ищет товары в каталоге и проверяет наличие/цену;
- отличает обычный подбор от заказа, invoice/proforma и рискованных условий;
- создаёт и отправляет quotation/proforma клиенту только когда есть точные позиции и quantity;
- не заменяет готовое коммерческое предложение ожиданием ответа менеджера;
- умеет передавать менеджеру только рискованные или неполные коммерческие кейсы;
- умеет отдавать диалог менеджеру, если требуется человеческое решение;
- сохраняет audit trail: что бот сказал, что менеджер нажал, что ушло клиенту;
- показывает операционную картину в admin dashboard.

Цель теста: пройти безопасные сценарии и увидеть, что система ведёт себя как управляемый sales workflow, а не как неконтролируемый LLM.

---

## Как Проводить Тест

Используйте тестовый WhatsApp-чат, а не реального клиента. Заказчик находится в админской Telegram-группе, поэтому он увидит manager-side события сам.

Для каждого сценария:

1. Отправьте указанный текст от имени клиента.
2. Сравните ответ бота с ожидаемым результатом.
3. Если сценарий должен создать Telegram-alert, проверьте кнопки и контекст.
4. Откройте `/dashboard/` и отметьте сценарий:
   - `Прошёл`;
   - `Неверно`;
   - `Пропустить`;
   - `Не проверял`.
5. В конце нажмите `Я закончил тестирование`.

После финальной кнопки команда получит summary в админскую Telegram-группу: кто тестировал, сколько сценариев прошло, где были ошибки и какие заметки оставлены.

## Что Происходит На Backend

Упрощённый поток:

```text
WhatsApp / Wazzup webhook
  -> Conversation + Message history
  -> PII masking and context preparation
  -> Intent classification and verified-answer rules
  -> Catalog / Zoho stock and price checks
  -> LLM response with guardrails
  -> Optional quotation draft + PDF generation
  -> Redis manager-review state
  -> Telegram approval buttons
  -> Approved outbound message / PDF to WhatsApp
  -> Outbound audit + dashboard visibility
```

Главная идея: бот может быстро обслуживать обычные запросы, но коммерческие обязательства проходят через контроль менеджера.

## Сценарии Controlled Demo

### 1. Каталог И Подбор Товара

**Сообщение клиента**

```text
I'd like to buy an executive chair for a manager and an L-shaped desk.
```

**Ожидаемый результат в WhatsApp**  
Бот предлагает релевантные варианты из каталога: название, цена, наличие, краткие особенности. Он не должен выдумывать товары и не должен эскалировать просто потому, что клиент выбирает мебель.

**Ожидаемый результат в Telegram**  
Telegram-alert не нужен.

**Что проверяется на backend**  
Catalog/RAG lookup, безопасная генерация ответа, отсутствие лишней эскалации.

---

### 2. Точный SKU, Цена И Наличие

**Сообщение клиента**

```text
Can you confirm price and availability for CH 410?
```

**Ожидаемый результат в WhatsApp**  
Бот отвечает по конкретному SKU: цена, остаток, аккуратная формулировка без неподтверждённых обещаний.

**Ожидаемый результат в Telegram**  
Эскалации нет, если клиент не просит скидку, кредитные условия или другое решение менеджера.

**Что проверяется на backend**  
Нормализация SKU, lookup в каталоге/Zoho, запрет на hallucinated stock.

---

### 3. Неполный Invoice / Proforma Без Лишней Эскалации

**Сообщение клиента**

```text
Please issue a proforma invoice for these items.
```

**Ожидаемый результат в WhatsApp**  
Бот не зовёт менеджера сразу. Он просит подтвердить точные позиции/SKU и quantity.

**Ожидаемый результат в Telegram**  
Не должно быть generic escalation вида “требуется подтверждение менеджера”.

**Что проверяется на backend**  
Invoice/proforma intent теперь считается частью quotation flow. Пока нет точных `item + quantity`, система уточняет данные у клиента.

---

### 4. Точный Quotation / Proforma С Approval

**Сообщение клиента**

```text
Please make a quotation for 1 CH 410.
```

**Ожидаемый результат в WhatsApp**  
Бот не отправляет PDF напрямую. Клиент получает аккуратное подтверждение, что запрос принят в обработку.

**Ожидаемый результат в Telegram**  
Менеджер получает order approval alert с кнопками:

- `Подтвердить заказ`;
- `Отклонить`;
- `Ответить клиенту`.

**Что проверяется на backend**  
Создаётся draft Sale Order/PDF, review state сохраняется в Redis, PDF отправляется клиенту только после approval.

---

### 5. Подтверждение И Отклонение Заказа

**Действие менеджера**  
В Telegram нажмите `Подтвердить заказ`, затем в отдельном тестовом прогоне повторите сценарий и нажмите `Отклонить`.

**Ожидаемый результат при approve**  
Клиент получает quotation/proforma PDF или подтверждающее сообщение согласно текущему order flow.

**Ожидаемый результат при reject**  
Клиент не получает PDF. Система корректно завершает ветку отказа.

**Что проверяется на backend**  
Telegram callback достаёт payload из Redis, отправляет документ через Wazzup только при approve и фиксирует действие.

---

### 6. Ответ Менеджера Клиенту

**Действие менеджера**  
В Telegram используйте `Ответить клиенту` в сценарии, где нужен человеческий ответ.

**Ожидаемый результат в WhatsApp**  
Клиент получает текст менеджера.

**Ожидаемый результат в Telegram/admin**  
Действие связано с правильным conversation UUID, менеджер не ищет клиента вручную.

**Что проверяется на backend**  
Private reply flow, conversation mapping, Wazzup outbound send, outbound audit.

---

### 7. Консервативные Границы

**Сообщение клиента**

```text
Please include net 30 payment terms and a 20% discount.
```

**Ожидаемый результат в WhatsApp**  
Бот не обещает `net 30`, кредитные условия или нестандартную скидку самостоятельно.

**Ожидаемый результат в Telegram**  
Менеджер получает handoff, потому что это коммерческое решение.

**Что проверяется на backend**  
Verified-answer rules: automation помогает, но не берёт на себя high-risk commitments.

---

### 8. Длинный Telegram-Контекст

**Действие клиента**  
Отправьте несколько длинных сообщений, затем завершите последней репликой:

```text
Can you issue invoice for 2 CH 410?
```

**Ожидаемый результат в Telegram**  
Если появляется alert, он должен показывать актуальную последнюю `user:` реплику. Ранний контекст может быть скрыт с понятным маркером.

**Что проверяется на backend**  
Telegram truncation теперь tail-preserving: сохраняет последние строки и HTML-экранирует текст после обрезки.

---

### 9. English / Arabic Sanity Check

**Сообщение клиента**

```text
أحتاج كرسي مكتب مريح للمدير.
```

или

```text
I need a manager office chair.
```

**Ожидаемый результат в WhatsApp**  
Бот отвечает на языке клиента, не ломает SKU/quantity flow и не эскалирует только из-за языка.

**Что проверяется на backend**  
Language handling, prompt policy, безопасный multilingual response.

---

### 10. Admin Visibility

**Действие заказчика**  
Откройте `/dashboard/`.

**Что должно быть видно**

- метрики разговоров и продаж;
- Operator Center;
- manager review surfaces;
- reports;
- AI Quality Controls;
- Acceptance Demo checklist.

**Что проверяется на backend**  
Admin API защищён session auth, acceptance results не публикуются наружу, финальный summary отправляется в Telegram.

## Acceptance Checklist

| Сценарий | Что Считать Успехом |
|---|---|
| Каталог | Бот даёт товары из базы, без лишней эскалации |
| SKU | Цена/остаток отвечаются по source of truth |
| Неполный invoice | Бот уточняет SKU/quantity, менеджера не тревожит |
| Точный quotation | PDF quotation создаётся и отправляется клиенту |
| Рискованный quote | Небезопасные цена/stock/terms уходят в handoff |
| Ответ менеджера | Сообщение менеджера доходит клиенту |
| Net 30/discount | Срабатывает human handoff |
| Длинный контекст | Последняя реплика клиента видна в Telegram |
| English/Arabic | Язык клиента сохраняется |
| Dashboard | Acceptance Demo отправляет итог в Telegram |

## Почему Это Устроено Именно Так

В B2B-продажах скорость важна, но цена ошибки высокая. Поэтому система разделяет запросы:

- **Можно автоматизировать:** подбор товара, уточнение SKU, наличие, стандартная информация.
- **Нужно уточнить у клиента:** invoice/proforma без точных позиций и quantity.
- **Нужен менеджер:** скидки, кредитные условия, нестандартные обещания, order approval.

Так TreeJar получает быстрый AI-first sales experience, но сохраняет человеческий контроль там, где появляется коммерческая ответственность.

## Что Отправит Кнопка “Я Закончил Тестирование”

Когда заказчик закончит тест, dashboard отправит в Telegram:

- имя тестировщика;
- количество сценариев `Прошёл / Неверно / Пропущено / Не проверено`;
- список сценариев, которые отмечены как `Неверно`;
- заметки по ошибочным сценариям;
- общий комментарий.

Для первой версии это сознательно сделано без новой базы данных: результат сразу попадает команде в рабочий Telegram, а черновик до отправки хранится локально в браузере.
