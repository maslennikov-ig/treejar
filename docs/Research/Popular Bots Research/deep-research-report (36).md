## Executive summary

Мой вывод: **не переписывать систему целиком в SaaS и не продолжать наращивать ad-hoc routing**, а **гибридизировать архитектуру**. Для вашего кейса лучше всего выглядит связка **LangGraph как stateful orchestration layer для диалога** и **Temporal как durable workflow layer для quotation / stock / CRM / PDF side effects**. Если команда готова заплатить более высокой ценой за более opinionated conversation engine и глубже перестроить саму логику диалога, то сильная альтернатива — **Rasa Pro / CALM** с его deterministic Flow Policy, conversation patterns и slot/flow semantics. citeturn29search3turn29search10turn28search4turn28search0turn31search16turn31search8

Ключевая мысль: ваши сбои выглядят не как “плохие промпты”, а как **ошибка разделения уровней состояния**. У вас сейчас текстовая память и tool routing частично подменяют собой бизнес-состояние. Для production sales assistant это неправильная граница. Состояние выбора товара, стадии закупки, набора уже показанных фото, quote-session и статуса handoff должно быть **явным, queryable, diffable и replayable**, а не только “лежать в истории сообщений”. Именно вокруг этого построены LangGraph persistence/checkpoints, Rasa Flow Policy и durable workflow-подход Temporal. citeturn35search15turn29search10turn31search16turn28search16

Самая практичная рекомендация для вас: **оставить custom domain kernel** — каталог, stock/price rules, Zoho CRM/Inventory mapping, PDF quotation logic, multilingual commercial phrasing, audit/event schema — и **заменить только orchestration/problematic glue**. В результате вы сократите сложность именно там, где сейчас ломается система: state continuity, recoverable quote flow, deterministic tool ordering и human handoff. citeturn28search0turn27search2turn31search3turn32search15

## Что на самом деле решает ваши падения

Ваши failure modes почти идеально раскладываются на четыре engineering-patterns.

Во-первых, **stable dialogue state**. Когда клиент сначала присылает фото, потом пишет текст, потом выбирает товар, а после этого внезапно переходит к quotation details, диалог должен жить не в prompt transcript, а в **каноническом state object**: `lead_stage`, `candidate_products`, `selected_products`, `shown_media_hashes`, `quote_session`, `missing_quote_fields`, `handoff_status`. В LangGraph это делается через explicit state + persistence/checkpoints; в Rasa CALM — через Flow Policy, flows, persisted slots, interruptions/resumptions и conversation patterns; в Botpress — через workflows и persisted tables. citeturn29search10turn29search3turn31search16turn31search0turn31search12turn32search0turn32search21

Во-вторых, **deterministic side effects**. Stock-check, price-check, CRM update, PDF generation и outbound send не должны быть “обычным tool call из LLM”. Это должны быть **durable business activities** с idempotency keys, retries, backoff и compensation. Temporal именно для этого и сделан: workflows and activities, RetryPolicy, visibility, reset/replay и Sagas/compensating actions для связанных distributed steps. Для quote/proforma/КП flow это полезнее любого “умного агента” само по себе. citeturn28search4turn27search2turn27search8turn28search0turn28search6turn28search13

В-третьих, **structured input в самом WhatsApp**, а не только free-form chat. Для product selection лучше использовать single-/multi-product messages и catalog messages, где пользователь выбирает product ID, а не “вот этот с третьей фотографии”. Для quotation data capture — WhatsApp Flows, где вы собираете обязательные поля в форме и получаете structured submission. Это напрямую снижает ваши конкретные баги: повторные фото после выбора, потеря контекста после quotation details и грязные переходы discovery → selection → quote. citeturn40search21turn40search13turn40search5turn40search2turn12search0turn12search3

В-четвёртых, **clean handoff**. Хороший handoff — это не просто “передать чат менеджеру”, а переключить owner conversation вместе с state packet: summary, selected SKUs, shown photos, pricing context, quote fields collected/missing, language, transcript window и reason for escalation. Rasa даёт built-in human handoff patterns, Botpress — HITL and Zoho SalesIQ handoff, Gupshup — Agent Assist/live-agent model, а respond.io документирует и обратную сторону проблемы: chained AI agents могут терять conversation context и повторять действия, если handoff не оформлен явно. citeturn31search7turn31search20turn32search15turn32search2turn40search7turn38search18

По reusable prompts/playbooks вывод тоже довольно жёсткий: **не ищите “волшебный sales prompt” как primary fix**. Наиболее полезные reusable assets — это stage-driven playbooks и deviation-handling patterns. Из open community артефактов самый близкий к вашему кейсу — `SalesGPT_fork`, который строит conversation по stages и прямо позиционируется как sales agent для text/voice и SMS/WhatsApp. Из более production-grade подходов — Rasa conversation patterns для corrections/interruptions/handoff и ready-made AI Agent instructions/actions у respond.io для handoffs, follow-ups и errors. citeturn19search0turn31search3turn38search10

## Рекомендованная целевая архитектура

Рекомендуемая целевая архитектура для вашего кейса — **не “один мощный агент”, а layered system**:
**канал и structured UX** → **stateful dialogue orchestrator** → **durable business workflow engine** → **custom domain tools/services** → **operator/handoff surface** → **audit/observability**. Такой разрез лучше совпадает и с LangGraph persistence model, и с Temporal durable workflow model, и с тем, как Rasa отделяет deterministic flows от language flexibility. citeturn29search10turn28search4turn31search16

Практически это выглядело бы так. Входящий webhook из Wazzup или альтернативного BSP сначала идёт в **channel normalizer**, который приводит text, image, catalog click, product-message reply и Flow submission к одному internal envelope. После этого **pre-router** определяет тип события: “vision-assisted discovery”, “product selection”, “quote data capture”, “resume existing quote”, “manager escalation” и так далее. Затем **LangGraph** держит canonical conversation state и решает, какой node активен: discovery, shortlist, selected-product memory, quote-subflow, handoff or resume. Любые side effects — вызов stock/price, create/update Zoho deal, generate PDF, send quotation, notify manager — исполняются уже не внутри LLM node, а как **Temporal activities/workflows**. citeturn29search3turn29search10turn28search4turn27search8

Самое важное изменение в архитектуре — ввести **объект quote session** как first-class entity. У него должны быть минимум: `quote_id`, `customer_id`, `language`, `selected_skus[]`, `price_context`, `stock_snapshot`, `filled_fields`, `missing_fields`, `last_completed_step`, `workflow_run_id`, `handoff_owner`, `manager_notes`, `pdf_artifact_id`. Когда клиент пишет “сделайте КП” после длинного discovery, бот не “вспоминает” из transcript, а **достаёт текущую quote session и resumes exact step**. Это архитектурно ближе к Rasa flows/slots и Temporal resumability, чем к обычному chat memory. citeturn31search0turn31search12turn29search10turn28search2

Для ваших двух самых болезненных кейсов я бы сделал жёсткие правила. **После подтверждения выбора товара** recommendation node больше не имеет права снова слать тот же SKU или тот же media set, пока stage не вернулся в re-opened discovery. И, наоборот, **после входа в quote subflow** бот не должен свободно ветвиться по catalog discovery; он должен либо park interruption и resume, либо передать summary менеджеру. LangGraph checkpoints и Rasa flow interruptions/resumptions дают для этого естественную механику. citeturn29search10turn31search12turn31search3

Если structured WhatsApp UX для вас доступен, я бы добавил ещё два сильных simplification levers. Первый — **product messages / catalog messages** для выбора товаров по product IDs; второй — **WhatsApp Flows** для сбора quotation fields. Это резко уменьшает долю “грязного текста”, который вы вынуждены интерпретировать LLM-ом, а значит — уменьшает сам объём routing exceptions. Если Wazzup не даёт удобного доступа к этим primitives, то это единственный слой, который я бы всерьёз рассмотрел к замене раньше, чем core Python system. citeturn40search21turn40search13turn40search5turn12search0turn12search3turn30search2turn30search17

Для operator/handoff слоя у вас есть развилка. Если текущие admin/operator tools приемлемы, оставьте их и просто передавайте richer state packet. Если же handoff UX слабый, то самый прагматичный путь — **не строить новый операторский inbox с нуля**, а взять готовую поверхность: Botpress + Zoho SalesIQ, respond.io или Gupshup Agent Assist. Для Zoho-centric организации **Botpress + Zoho SalesIQ HITL** выглядит особенно практично. citeturn32search2turn32search15turn40search7turn40search14

## Что оставить custom и что заменить

**Что оставить custom.** Я бы точно не отдавал наружу ваш **product domain kernel**: каталожный search/ranking, сопоставление SKU/variant/media, stock/price business rules, discount approvals, quote/proforma/PDF templates, mapping в Zoho CRM/Inventory и ваш сквозной audit event model. Это то место, где generic platforms дают только primitives — tools, workflows, KB, tables, CRM connectors — но не знают ваших коммерческих правил и исключений. Здесь ценность уже накоплена именно в вашем коде. citeturn32search0turn27search0turn26search1turn40search2

Точно так же стоит оставить custom **multilingual commercial layer**: как именно вы продаёте офисную мебель на английском, русском и арабском, как оформляете КП, как формулируете уточнения по quantity, lead time, fabric, finish, delivery, Incoterms и payment terms. Готовые platforms поддерживают multilingual AI, но это не заменяет ваш domain-specific sales playbook. Лучшее, что можно сделать, — не заменять это, а положить поверх explicit stage machine и quote session. citeturn31search5turn38search0turn38search1

**Что заменить или упростить.** Первое — ваш текущий **custom routing layer**. Его стоит заменить на LangGraph state graph или Rasa flows. Второе — **quote recovery logic**: вынести её из условных веток в durable workflow engine с idempotent activities. Третье — **примитивные glue-automations** вокруг CRM, notify, attachments, template sends: это хорошо отдаётся n8n. Четвёртое — **operator handoff surface**, если он у вас слабый: тут уместен частичный buy вместо build. citeturn29search3turn31search16turn28search4turn39search0turn32search15

Если сформулировать совсем коротко, то целевой split такой:
**custom** = pricing, stock, catalog intelligence, quote documents, compliance/audit, multilingual sales policy;
**framework/platform** = dialogue orchestration, structured handoff, retry/recovery engine, low-risk CRM glue, structured WhatsApp UX. citeturn28search0turn31search16turn39search12turn40search21

## План PoC на неделю

Я бы делал PoC не как “построить нового бота”, а как **replay-based proof**, который показывает, что 4–5 конкретных failure modes больше не ломают систему. PoC должен прогоняться на реальных анонимизированных production transcripts и оценивать только то, что реально болит: duplicate product photos, lost quote context, failed quote recovery, bad tool ordering и broken manager escalation. LangGraph checkpoints, Temporal retries/sagas и Rasa-style deterministic flow semantics дают хороший baseline для такого сравнения. citeturn29search10turn28search15turn31search16

**День первый** — формализовать canonical state schema и acceptance pack из 20–30 реальных эпизодов. Отдельно пометить: “photo first”, “photo repeat after selection”, “quote after long discovery”, “resume quote after interruption”, “manager escalation”. Выход дня — Excel/JSON test suite и list of invariants.

**День второй** — собрать минимальный LangGraph прототип с четырьмя stages: discovery, shortlist, selected-product memory, quote-subflow. Никаких красивых промптов; только explicit state transitions, state store и traceability.

**День третий** — подключить catalog lookup, image-aware candidate extraction и suppression rules: после `selected_skus[]` bot не имеет права повторять выбранные позиции без явного re-open.

**День четвёртый** — вынести quote path в отдельный Temporal workflow: `reserve_context → fetch_stock → fetch_price → draft_offer → create_pdf → update_crm → send_or_escalate`. Каждая activity получает idempotency key и explicit failure class.

**День пятый** — добавить human handoff packet: summary + selected SKUs + filled/missing quote fields + current step + recent transcript chunk.

**День шестой** — прогнать acceptance pack и измерить diff against current system: сколько duplicate-photo cases осталось, сколько quote sessions resume correctly, сколько errors требуют manual recovery.

**День седьмой** — decision review: либо идти в full migration to LangGraph+Temporal, либо переключаться на Rasa-first path, если окажется, что основной выигрыш даёт именно conversation engine, а не Python-preserving architecture.

Критерии успеха PoC я бы поставил такими: **нулевые повторные рекомендации уже выбранного SKU**, **resumability quote flow после interruption/restart**, **traceable history каждого tool call и state transition**, **human handoff с полным state packet**, **сокращение custom routing rules минимум на треть уже в пилоте**. Это не интернет-факты, а мой engineering target для принятия решения после PoC.

## Красные флаги и поисковое приложение

**Красные флаги.** Первый — **не делать Flowise primary runtime для customer-facing sales assistant**. У него есть visual builder, Agentflow, HITL, tracing и analytics, но за 2025–2026 surfaced и серьёзные security advisories, включая RCE в CustomMCP и CSV-agent prompt-injection/RCE class issues, а также recurring tool-calling inconsistencies и even deprecating Agentflow V1. Для внутренних прототипов это всё ещё интересный инструмент; для вашего production sales path — слишком рискованно. citeturn33search0turn33search1turn33search2turn33search13turn33search15turn33search17turn33news38turn0search12

Второй — **не делать CrewAI основным customer-facing runtime**, несмотря на красивые Flows/HITL/state docs. Документация у него хорошая, но community signal по function/tool calling слишком неровный: GPT-5 incompatibilities, native function-calling bugs и repeated tool invocation. Для internal task orchestration это терпимо; для продаж, где quote flow должен быть детерминированным, это плохой компромисс. citeturn34search0turn34search11turn34search2turn34search6turn34search20

Третий — **не брать LlamaIndex или AutoGen как ваш primary sales orchestrator именно сейчас**, если цель — быстро снять production pain. Четвёртый — **не использовать n8n как единственный source of truth для multi-turn quotation context**. Он великолепен как automation fabric, но official docs прямо говорят, что memory у Tools Agent не persists between sessions by default. Добавьте сюда documented rough edges в WhatsApp template nodes и recent high-severity vulnerabilities — и становится ясно, что n8n должен быть вокруг core brain, а не вместо него. citeturn39search6turn39search7turn39search16turn37news39turn37news40turn37news38

По surfaced docs он закрывает messaging/templates/webhooks/CRM quite adequately, но не выглядит как очевидный стратегический слой для Flows/catalog-driven UX. Для первой фазы — оставить можно; для следующей — лучше отделить channel provider decision от dialogue/state decision. citeturn30search2turn30search5turn30search17turn30search6
