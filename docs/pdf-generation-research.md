# Исследование: Генерация сложных коммерческих предложений (PDF)

В рамках 5-й недели проекта **Noor AI Seller** нам необходимо реализовать возможность создания сложных коммерческих предложений (КП/SaleOrder) в формате PDF. Документ должен включать: шапку, информацию о клиенте, таблицу товаров с изображениями и артикулами, а также итоговую калькуляцию с учетом налогов (VAT 5%).

Мы рассмотрим два принципиальных подхода: 
1. Встроенные возможности Zoho Inventory (шаблоны PDF).
2. Генерация PDF "с нуля" на стороне Python/FastAPI.

---

## Вариант 1: Использование встроенного шаблонизатора Zoho Inventory

Zoho Inventory обладает мощным встроенным редактором шаблонов для выгрузки документов (Invoices, Sale Orders).

**Как это работает:**
Наше FastAPI приложение только формирует заказ (собирает ID клиента и список SKU с их количеством) и отправляет `POST /api/v1/inventory/sale-orders/`.  
Вся визуальная часть шаблонизируется внутри интерфейса Zoho. Zoho API позволяет выгрузить готовый PDF по ID заказа.

**Плюсы:**
- **Поддержка картинок товаров:** Zoho Inventory нативно поддерживает вывод изображения товара в шаблоне PDF (которое берется из карточки товара в базе).
- **Единый источник истины:** Менеджеры могут в любой момент зайти в Zoho, внести ручные правки в сгенерированный заказом ИИ-бота Sale Order, и заново скачать актуальный PDF.
- **Поддержка налогов и калькуляции (VAT 5%):** Линии товаров, скидки, и налоги высчитываются внутри Zoho. Нам не придется хардкодить бухгалтерскую математику на Python.
- **Никакой нагрузки на наш сервер:** Не нужно рендерить HTML, держать запущенный браузер (Playwright) или тянуть тяжелые библиотеки.
- **Легкость настройки визуала:** Дизайн КП и логотипы можно легко настроить мышкой в встроенном drag-and-drop редакторе Zoho (или загрузить свой HTML/CSS шаблон). 

**Минусы:**
- Ограничения встроенного редактора Zoho: если нам потребуется дизайн, который принципиально не вписывается в их HTML-таблицы, это может стать блокером.
- Требуется правильно загружать или связывать картинки с товарами внутри самой CRM/Inventory.

## Вариант 2: Генерация PDF в Python/FastAPI

В этом подходе мы только тянем данные о товаре и ценах из БД/API, а дальше берем HTML-шаблон (Jinja2), рендерим его и перегоняем в PDF на сервере.

Сравнение библиотек:
1. **WeasyPrint**  
   - *Плюсы:* Отличная поддержка HTML/CSS, не требует запуска полноценного браузера, идеален для статической генерации отчетов.
   - *Минусы:* Требует системную библиотеку Pango/Cairo. Верстка таблиц с разрывами страниц иногда ломается.
   
2. **Playwright Pypeteer (Headless Browser)**  
   - *Плюсы:* Pixel-perfect рендеринг: PDF будет выглядеть ровно на 100% так же, как веб-страница со всеми современными CSS.
   - *Минусы:* Очень ресурсоемкий. Запуск инстанса браузера внутри Docker контейнера (нужно доустанавливать системные шрифты и зависимости Chromium). Большое потребление RAM.
   
3. **ReportLab**  
   - *Плюсы:* Самая стабильная, быстрая и мощная библиотека.
   - *Минусы:* Рисует PDF через "холст" и координаты (или свой язык разметки RML), а не через HTML/CSS. Очень тяжело внедрять дизайн по макету и рисовать красивые таблицы с картинками "вручную".

**Плюсы подхода "с нуля":**
- Полный контроль над дизайном до единого пикселя.
- Возможность вставлять динамические элементы, которых нет в Zoho (доп. графики, специфичные блоки).

**Минусы подхода "с нуля":**
- Нужно самостоятельно делать верстку HTML/CSS для генерации PDF-страниц с постраничной разбивкой.
- При скачивании картинок товаров по URL для вставки в PDF скорость значительно упадет (нужно перетягивать их в генератор).
- Финансовая логика: придется самостоятельно высчитывать скидки и налоги в Python. 
- Любое изменение в тексте договора или реквизитов (например, смена адреса компании) потребует деплоя новой версии кода (или редактирования HTML в репозитории), а не простого изменения через админку Zoho.

## Вердикт и архитектурное решение

**Рекомендуемый подход: Вариант 1 (Использование Zoho Inventory Templates)**

Для проекта Noor AI Seller использование Zoho Inventory для финальной генерации PDF — это **наилучший выбор по соотношению качество/трудозатраты/поддерживаемость**. 

Поскольку Zoho нативно поддерживает вывод изображений (`Product Image`) в таблице Quotation/Sale Order, мы можем легко создать брендированный шаблон в панели администратора Zoho. 

**Flow работы бота на 5-й неделе будет таким:**
1. Бот LLM опрашивает клиента и формирует заказ (собирает SKU, количество, цены).
2. Вызывает `create_quotation`, который маппится в `POST /api/v1/inventory/sale-orders`.
3. В Zoho Inventory автоматически появляется Sale Order. Все налоги и суммы рассчитываются на стороне Zoho.
4. Zoho подтягивает картинки товаров в сформированный документ по заранее настроенному кастомному шаблону.
5. Наше приложение запрашивает выгрузку этого Sale Order в формате PDF через API Zoho.
6. Полученный PDF отправляется клиенту в WhatsApp с помощью коннектора Wazzup. 

Это решение исключит расхождения между отправленным в WhatsApp PDF-файлом и реальным заказом, сохраненным в CRM/Inventory, и позволит заказчику легко менять дизайн КП без привлечения программистов.
